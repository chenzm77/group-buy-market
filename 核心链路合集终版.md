# 拼团电商核心服务统一架构文档

> 本文档用于指导 LLM 按 DDD 分层架构生成拼团电商的核心玩法服务。核心服务包括：库存服务、拼团服务、订单服务、支付服务、履约服务、退单服务。本文档不按单个服务孤立展开，而是从统一业务状态机、跨服务协作、核心链路、幂等与补偿、普通订单与拼团订单抽象等角度整体描述。

---

## 1. 项目定位与核心复杂度

本项目是拼团电商系统。与红包、券、积分等营销能力相比，拼团不是一个简单的外挂式营销工具，而是会深度侵入订单、库存、支付、履约、退单等状态机的核心玩法。

普通电商订单的主链路通常是：

```text
下单锁库存 → 支付 → 确认扣减库存 → 创建履约 → 发货 → 收货 → 售后
```

拼团订单的主链路在普通订单基础上增加了团状态机和成员状态机：

```text
下单锁库存/占团位 → 创建订单 → 支付 → 成员进入待成团 → 判断团是否成功 → 成团后订单进入待发货 → 履约
```

因此，拼团订单不是普通订单之外的另一个完全独立模型，而是普通订单状态机的扩展。普通订单可以看作拼团订单在“无需组团、支付即满足履约条件”场景下的子集。

## 2. 总体架构原则

### 2.1 DDD 分层原则

每个核心服务都采用统一的 DDD 六模块结构：

```text
{service}/
├── {service}-api            # RPC 接口、DTO、Response
├── {service}-types          # 枚举、常量、异常
├── {service}-domain         # 聚合、实体、值对象、仓储接口、领域服务、领域事件
├── {service}-infrastructure # DAO、PO、RepositoryImpl、Redis、MQ、外部适配
├── {service}-app            # 应用服务：事务、用例编排、DTO转换
└── {service}-trigger        # Dubbo RPC、HTTP Controller、MQ Consumer、定时任务
```

依赖方向必须满足：

```text
trigger → app → domain ← infrastructure
   │       │      │           │
   └────── api    types ←─────┘
```

核心规则：

1. 业务规则只写在 domain。
2. app 只做用例编排、事务边界、DTO/Entity 转换、领域服务调用、事件发布。
3. infrastructure 只做技术适配，不写业务判断。
4. trigger 只做协议适配、异常包装、MQ 消费入口、定时任务入口。
5. 跨服务调用必须通过 domain 层定义的 ACL 接口，由 infrastructure 实现 Dubbo/MQ 适配。

### 2.2 跨服务协作模式

核心链路使用两类通信方式：

1. Dubbo 同步调用：用于必须立即知道结果的强前置动作，例如下单时锁库存、占团位、创建支付单时读取订单。
2. RocketMQ 异步消息：用于状态最终推进、补偿、延时关单、延时关团、支付成功通知、履约创建等最终一致性动作。

下单主链路是一个手写 Saga 风格的分布式事务：

```text
订单服务作为主编排者
  → 同步调用库存服务锁库存
  → 同步调用拼团服务占团位（拼团订单）
  → 本地创建订单
  → Outbox/事务消息发送延时关单、订单创建成功事件
```

与 Seata Saga 不同：

- Seata Saga 使用状态机中间件保存状态图并自动调用补偿。
- 本项目由订单服务显式编排主流程，失败时主动调用补偿接口。
- 同时通过延时消息、定时任务、幽灵扫描任务保证最终一致性。

## 3. 核心服务职责边界

### 3.1 库存服务 inventory

库存服务负责 SKU 库存与活动限额的锁定、确认扣减、释放、退回。它不关心订单业务含义，只根据调用场景执行库存状态转移。

核心职责：

- SKU 库存管理。
- 活动限额管理。
- 下单锁库存。
- 支付且满足履约条件后的确认扣减。
- 订单取消/超时未支付释放锁定库存。
- 退款/退单退回已售库存。
- 库存流水与下单扣减幂等记录。
- 幽灵扣减扫描与补偿。

库存字段：

```text
total_stock     总库存
available_stock 可用库存
locked_stock    已锁库存，下单后未最终售出
sold_stock      已售库存，支付且业务最终确认后计入
```

恒等式：

```text
total_stock = available_stock + locked_stock + sold_stock
```

库存状态转移：

```text
下单锁库存：available_stock - quantity，locked_stock + quantity
支付确认/成团确认：locked_stock - quantity，sold_stock + quantity
超时未支付/取消订单：locked_stock - quantity，available_stock + quantity
退款/退单：sold_stock - quantity，available_stock + quantity
```

活动限额字段：

```text
total_quota     活动总限额
available_quota 活动可用限额
locked_quota    活动已锁限额
used_quota      活动已使用限额
```

活动限额状态转移：

```text
拼团下单：available_quota - quantity，locked_quota + quantity
支付且成团/最终确认：locked_quota - quantity，used_quota + quantity
未支付取消：locked_quota - quantity，available_quota + quantity
退单：used_quota - quantity，available_quota + quantity
```

普通订单不涉及 activity_quota，拼团订单同时操作 SKU 库存和活动限额。

### 3.2 拼团服务 groupbuy

拼团服务负责活动下的团、团成员、团排行榜、团状态推进。它不负责订单支付、不负责库存真实扣减、不负责履约。

核心职责：

- 创建团。
- 加入团。
- 占团位。
- 订单创建后绑定 orderId。
- 支付成功后成员进入组团中。
- 判断是否成团。
- 成团后通知订单服务推进订单。
- 超时关团。
- 取消订单释放团位。
- 退单后成员状态流转为已退款，但通常不回滚团成功状态。
- 维护活动详情页展示需要的拼团统计与 Top N 待成团团列表。

拼团服务是拼团玩法的核心状态机所在地。

### 3.3 订单服务 order

订单服务是下单主链路的编排者，也是订单状态机的事实源。

核心职责：

- 接收普通订单和拼团订单的统一下单请求。
- 参数预校验。
- requestId 幂等。
- 编排库存锁定。
- 编排拼团占位。
- 创建订单。
- 发送延时关单消息。
- 接收支付成功事件并抢订单状态。
- 结算成功后通知库存、拼团。
- 成团后推进订单到待发货。
- 取消订单、超时关单。
- 接收关团导致的订单取消。
- 为退单服务提供订单状态推进接口。

订单服务不直接对接第三方支付，不直接管理物流，不直接改库存表和拼团表。

### 3.4 支付服务 payment

支付服务是支付限界上下文，负责对接第三方支付能力。

核心职责：

- 创建支付流水。
- 调用支付宝电脑网站支付接口生成支付表单/支付链接。
- 处理支付宝异步回调。
- 验签。
- 更新支付流水状态。
- 通过 MQ 通知订单服务支付成功。
- 订单服务无法承接支付时，执行原路退款。
- 退单服务编排退款时，调用支付服务退款接口。

支付服务与订单服务分离的原因：

- 支付有独立状态机。
- 支付涉及渠道、密钥、签名、验签、回调、退款。
- 支付渠道变化不应污染订单服务。
- 支付成功事实以支付服务为准，订单后续状态以最终一致方式推进。

### 3.5 履约服务 fulfillment

履约服务负责订单进入可发货状态后的履约单、物流信息和收货状态。

核心职责：

- 创建履约单。
- 维护履约状态：待发货、已发货、已签收、已取消、已退单。
- 运营手动填写物流公司和物流单号。
- 用户确认收货。
- 为退单服务提供履约状态推进接口。

本项目初版不直接对接物流公司，只做到运营发货、用户确认收货。

### 3.6 退单服务 refund

退单服务负责售后退单申请、审批和退单流程编排。

核心职责：

- 用户申请退单。
- 生成退单申请记录。
- 运营审批。
- 退单流程编排。
- 调用订单服务推进订单退单状态。
- 调用库存服务退回库存。
- 调用拼团服务更新成员退款状态。
- 调用支付服务退款。
- 调用履约服务关闭或更新履约单。

退单服务是售后逆向链路的编排者。

## 4. 普通订单与拼团订单的统一抽象

### 4.1 订单类型

订单服务必须支持两类订单：

```text
NORMAL      普通订单
GROUP_BUY   拼团订单
```

订单表中应有：

```text
order_type       订单类型：NORMAL/GROUP_BUY
activity_id      活动ID，普通订单为空
team_id          团ID，普通订单为空
team_member_id   团成员ID，普通订单为空
request_id       下单请求ID，全链路幂等键
```

### 4.2 下单接口统一设计

订单服务对外暴露统一下单接口：

```text
createOrder(CreateOrderCommandDTO command)
```

关键入参：

```text
requestId
userId
skuId
spuId
quantity
addressId
orderType
activityId      拼团订单必填
teamId          参团时必填，开团时为空
groupAction     OPEN_TEAM/JOIN_TEAM，拼团订单必填
```

### 4.3 用策略消除接口通用性难题

普通订单和拼团订单的差异不应通过 app 层大量 if-else 扩散。推荐在订单 domain 或 app 中定义下单策略：

```text
OrderPlaceStrategy
├── NormalOrderPlaceStrategy
└── GroupBuyOrderPlaceStrategy
```

统一模板：

```text
1. 订单服务接收统一 createOrder 请求
2. 根据 orderType 选择策略
3. 执行策略的 preValidate
4. 执行策略的 reserveResource
5. 创建订单聚合
6. 执行策略的 afterOrderCreated
7. 发送延时关单消息
8. 返回 orderId
```

普通订单策略：

```text
preValidate：校验商品、SKU、地址、价格、库存
reserveResource：只调用库存服务 lockInventory，不传 activityId
createOrder：创建 NORMAL 订单
payload：无需 teamId/teamMemberId
```

拼团订单策略：

```text
preValidate：校验商品、SKU、地址、活动、用户参团资格、团队状态
reserveResource：调用库存服务 lockInventory，同时扣 SKU 库存与活动限额；调用拼团服务 occupyTeamSeat
createOrder：创建 GROUP_BUY 订单，写入 teamId/teamMemberId
payload：需要 activityId/teamId/teamMemberId/groupAction
```

普通订单是拼团订单的子集：

- 无活动限额。
- 无团位占用。
- 无待成团状态。
- 支付成功后直接进入待发货。

## 5. 全局状态机

### 5.1 订单状态机

订单状态建议：

```text
CREATED          已创建，待支付
WAIT_GROUP       已支付，待成团，仅拼团订单使用
WAIT_SHIP        待发货
SHIPPED          已发货
RECEIVED         已收货
CANCELLED        已取消，未支付取消/超时关单
REFUND_APPLYING  退单申请中
REFUNDING        退单处理中
REFUNDED         已退单
CLOSED           已关闭，终态兜底
```

普通订单主状态转移：

```text
CREATED → WAIT_SHIP → SHIPPED → RECEIVED
CREATED → CANCELLED
WAIT_SHIP/SHIPPED/RECEIVED → REFUND_APPLYING → REFUNDING → REFUNDED
```

拼团订单主状态转移：

```text
CREATED → WAIT_GROUP → WAIT_SHIP → SHIPPED → RECEIVED
CREATED → CANCELLED
WAIT_GROUP → CANCELLED       团超时失败导致取消/退款
WAIT_GROUP/WAIT_SHIP/SHIPPED/RECEIVED → REFUND_APPLYING → REFUNDING → REFUNDED
```

订单状态抢占原则：

- 支付结算和超时关单天然冲突。
- 谁先 CAS 更新订单状态，谁获得后续处理权。
- 例如支付成功只能从 CREATED 改到 WAIT_GROUP 或 WAIT_SHIP。
- 超时关单只能从 CREATED 改到 CANCELLED。

### 5.2 支付流水状态机

支付流水状态：

```text
INIT       已创建，准备调渠道
PAYING     已获得支付链接/等待支付
SUCCESS    支付成功
FAILED     支付失败
CLOSED     已关闭
REFUNDING  退款中
REFUNDED   已退款
```

状态转移：

```text
INIT → PAYING → SUCCESS
INIT/PAYING → CLOSED
SUCCESS → REFUNDING → REFUNDED
```

### 5.3 团状态机

团状态：

```text
WAIT_LEADER_ORDER  团已创建，等待团长订单创建
FORMING            团长已支付或已有成员支付，正在组团
SUCCESS            已成团
FAILED_TIMEOUT     超时未成团
CANCELED           团长下单失败或团取消
```

状态转移：

```text
WAIT_LEADER_ORDER → FORMING
FORMING → SUCCESS
WAIT_LEADER_ORDER/FORMING → FAILED_TIMEOUT
WAIT_LEADER_ORDER → CANCELED
```

团成功后原则上不因个别用户退单而回滚成团状态。

### 5.4 团成员状态机

团成员状态：

```text
PENDING_ORDER   已占团位，等待订单创建
WAIT_PAY        订单已创建，待支付
GROUPING        已支付，待成团
GROUP_SUCCESS   已成团
CANCELED        已取消
REFUNDED        已退款
```

状态转移：

```text
PENDING_ORDER → WAIT_PAY → GROUPING → GROUP_SUCCESS
PENDING_ORDER/WAIT_PAY → CANCELED
GROUPING/GROUP_SUCCESS → REFUNDED
```

### 5.5 库存状态机

库存无单独业务状态，依靠数量字段和流水表达状态转移。

```text
LOCK 下单锁定
CONFIRM 支付确认/成团确认
RELEASE 未支付取消释放
REFUND 退单退回
COMPENSATE 下单失败补偿
```

## 6. 核心链路一：商品详情与确认页

### 6.1 商品详情页

商品详情页请求打到商品服务。商品服务聚合返回前端所需的大 DTO。

商品服务同步查询：

```text
1. 用户服务：查询用户地址列表
2. 商品服务本地：SPU、SKU、Activity、ActivityItem
3. 如果 activity 不为空：调用拼团服务 serveForShopDetail
```

拼团服务 serveForShopDetail 返回：

```text
successTeamCount       已成团数量
successPeopleCount     已拼成人数
waitingPeopleCount     正在拼团人数
topWaitingTeams        Top N 待成团团列表
```

Top 待成团团列表使用 Redis ZSet：

```text
key：activity:team:rank:{activityId}
score：还差几人支付/成团，越小优先级越高
value：teamId
```

选择“还差几人”作为优先级原因：

- 更容易展示临门一脚的团。
- 用户参团后更容易快速成团。
- 避免把用户引导到马上过期、体验差的濒死团。

### 6.2 确认页

确认页是前端浮层，复用商品详情数据。打开确认页时可以再次调用商品详情接口，或调用轻量校验接口。

轻量确认接口：

```text
GET /order/confirm/check?skuId=xxx&addressId=xxx&activityId=xxx&teamId=xxx
```

后端检查：

```text
SKU 实时价格
SKU 实时库存
活动是否有效
团队是否可加入
用户是否满足资格
地址是否有效
```

确认页只是提升体验，不代替下单接口最终校验。

## 7. 核心链路二：统一下单链路

### 7.1 requestId 设计

用户点击下单时，前端生成 UUID 格式 requestId。一次下单请求全链路只使用一个 requestId。

requestId 用途：

- 前端重复点击使用同一个 requestId。
- 前端超时重试使用同一个 requestId。
- 订单服务创建订单幂等。
- 库存服务锁库存幂等。
- 拼团服务占团位幂等。
- 失败补偿定位。

前端行为：

- 第一次点击生成 requestId。
- 下单按钮置灰，阻止重复操作。
- 收到最终响应后清除 requestId 或刷新页面。

后端入口：

- 使用短 TTL 分布式锁防止快速重复点击。
- 分布式锁只挡大部分并发，不作为唯一正确性保证。
- 每个服务节点内部必须自带幂等与并发控制。

### 7.2 下单总流程

```text
用户提交下单
  → 订单服务 createOrder
    → requestId 短 TTL 锁
    → 校验商品、SKU、地址、价格、活动、团队
    → 同步调用库存服务 lockInventory
    → 如果是拼团订单，同步调用拼团服务 occupyTeamSeat
    → 本地创建订单
    → 发送延时关单消息
    → 如果是拼团订单，发送订单创建成功消息/同步通知拼团绑定 orderId
    → 返回 orderId
```

### 7.3 下单断点续跑

场景一：请求 A 扣完库存后崩溃，请求 B 重试：

```text
B 调库存：requestId 已存在，视为已锁库存
B 调拼团：未做过，继续占团位
B 创建订单：未做过，继续创建
```

场景二：请求 A 占团位成功后崩溃，请求 B 重试：

```text
B 调库存：幂等跳过
B 调拼团：requestId 已存在，返回 teamId/teamMemberId
B 创建订单：继续创建
```

场景三：请求 A 全部完成但响应丢失，请求 B 重试：

```text
B 调库存：幂等跳过
B 调拼团：幂等返回
B 创建订单：requestId 已存在，查出 orderId 返回
```

## 8. 库存服务详细设计

### 8.1 主要聚合

```text
Stock 聚合
- skuId
- totalStock
- availableStock
- lockedStock
- soldStock
- lock(quantity)
- confirm(quantity)
- release(quantity)
- refund(quantity)

ActivityQuota 聚合
- activityId
- skuId
- totalQuota
- availableQuota
- lockedQuota
- usedQuota
- lock(quantity)
- confirm(quantity)
- release(quantity)
- refund(quantity)
```

### 8.2 核心表

SKU 库存表：

```sql
CREATE TABLE t_sku_stock (
    id BIGINT NOT NULL,
    sku_id BIGINT NOT NULL,
    total_stock INT NOT NULL DEFAULT 0,
    available_stock INT NOT NULL DEFAULT 0,
    locked_stock INT NOT NULL DEFAULT 0,
    sold_stock INT NOT NULL DEFAULT 0,
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    UNIQUE KEY uk_sku_id (sku_id)
```

### 8.3 lockInventory 接口

入参：

```text
requestId
orderType
skuId
activityId
quantity
```

流程：

```text
1. 插入 t_inventory_deduction，status=PROCESSING
2. 插入失败：根据 requestId 查询记录
   - SUCCESS：返回成功
   - PROCESSING：可短暂等待或返回处理中，由调用方重试
   - FAILED/CANCELED/COMPENSATED：按业务决定返回失败或允许接管
3. 条件更新 SKU 库存：available_stock >= quantity
4. 如果是拼团订单，条件更新 activity_quota：available_quota >= quantity
5. 两个更新都成功：状态改 SUCCESS，删除库存缓存
6. 任一失败：已扣部分回滚补回，状态改 FAILED，删除库存缓存
```

条件更新示例：

```sql
UPDATE t_sku_stock
SET available_stock = available_stock - ?, locked_stock = locked_stock + ?
WHERE sku_id = ? AND available_stock >= ?;
```

### 8.4 confirmInventory 接口

支付成功后是否立即确认库存，取决于订单类型：

- 普通订单：支付成功后订单直接满足履约条件，可以确认库存。
- 拼团订单：支付成功后只是成员进入待成团；真正确认库存可以在成团成功时执行。

为了统一，建议订单服务在“订单进入 WAIT_SHIP”时调用库存确认接口。这样：

```text
普通订单：支付成功 → WAIT_SHIP → confirmInventory
拼团订单：成团成功 → WAIT_SHIP → confirmInventory
```

接口使用 orderId + CONFIRM 做幂等。

### 8.5 releaseInventory 接口

用于未支付取消、超时关单、团长下单失败等场景。

幂等键：

```text
orderId + RELEASE
```

如果订单尚未创建但库存已锁定，由幽灵扫描任务用 requestId 做补偿。

### 8.6 refundInventory 接口

用于已支付后退单。

幂等键：

```text
refundId + REFUND
```

状态转移：

```text
sold_stock - quantity
available_stock + quantity
```

拼团订单同步退回 activity_quota：

```text
used_quota - quantity
available_quota + quantity
```

### 8.7 Redis 缓存

Redis 只用于读缓存，不参与写侧预扣减。

```text
sku:stock:{skuId}                       → available_stock
activity:quota:{activityId}:{skuId}     → available_quota
```

写操作成功后删除缓存，读请求按 Cache-Aside 回源数据库。

### 9.3 occupyTeamSeat 接口

入参：

```text
requestId
userId
activityId
skuId
teamId
groupAction OPEN_TEAM/JOIN_TEAM
```

开团流程：

```text
1. 根据 requestId 查询 team_member，存在则幂等返回
2. 创建 team，status=WAIT_LEADER_ORDER，locked_count=1，paid_count=0
3. 创建 team_member，status=PENDING_ORDER，role=LEADER
4. 发送延时关团消息
5. 写入/更新活动 team ZSet
6. 返回 teamId/teamMemberId
```

参团流程：

```text
1. 根据 requestId 查询 team_member，存在则幂等返回
2. 校验 team 状态为 FORMING 或 WAIT_LEADER_ORDER，且 locked_count < target_count
3. 条件更新 team.locked_count + 1
4. 插入 team_member，status=PENDING_ORDER，role=MEMBER
5. 如果插入失败，回滚 locked_count
6. 更新活动 team ZSet
7. 返回 teamId/teamMemberId
```

### 9.4 bindOrder 接口

订单创建成功后，订单服务通知拼团服务绑定 orderId。

入参：

```text
requestId
teamId
teamMemberId
orderId
```

状态转移：

```text
PENDING_ORDER → WAIT_PAY
```

### 9.5 memberPaid 接口

订单支付成功后，订单服务通知拼团服务。

状态转移：

```text
team_member: WAIT_PAY → GROUPING
team: paid_count + 1
```

如果 `paid_count == target_count`：

```text
team: FORMING → SUCCESS
team_member: GROUPING → GROUP_SUCCESS
发送 GroupTeamSuccessEvent 给订单服务
从活动 team ZSet 移除该 team
```

如果未成团：

```text
team: WAIT_LEADER_ORDER → FORMING
更新活动 team ZSet score = target_count - paid_count
```

### 9.6 cancelMember 接口

用于未支付取消/超时关单。

状态转移：

```text
PENDING_ORDER/WAIT_PAY → CANCELED
team.locked_count - 1
```

如果取消后团没有有效成员，team 可转 CANCELED。

### 9.7 refundMember 接口

用于已支付后退单。

状态转移：

```text
GROUPING/GROUP_SUCCESS → REFUNDED
team.paid_count - 1
```

原则：

- 团已 SUCCESS 后，不因单个成员退款而回滚团状态。
- 如果团仍 FORMING，退款会降低 paid_count 并更新 ZSet。

### 9.8 timeoutCloseTeam

开团时发送延时消息。到期后：

```text
1. 查询 team
2. 如果 team 未 SUCCESS，CAS 改为 FAILED_TIMEOUT
3. 查询该 team 下所有非终态 team_member
4. 给订单服务发送 TeamTimeoutCancelOrderEvent
5. 订单服务逐个取消关联订单
6. 从活动 team ZSet 移除该 team
```

### 10.3 createOrder

流程：

```text
1. requestId 短 TTL 分布式锁
2. 根据 requestId 查询订单，存在则直接返回 orderId
3. 读取最新商品/SKU/价格/地址/活动/团队信息
4. 根据 orderType 选择下单策略
5. 调库存服务 lockInventory
6. 拼团订单调拼团服务 occupyTeamSeat
7. 本地事务创建订单，唯一键 uk_request_id 保证幂等
8. 写 Outbox：OrderCreatedEvent、OrderCloseDelayEvent
9. 返回 orderId
```

如果第 5 或第 6 成功后第 7 失败：

- 同步补偿能做则立即补偿。
- 同时依赖库存服务 t_inventory_deduction 幽灵扫描。
- 拼团服务 PENDING_ORDER 长时间未绑定订单也应由扫描任务取消。

### 10.4 支付成功消费

支付服务发送 PaymentSuccessEvent，订单服务消费。

流程：

```text
1. 根据 orderId 查询订单
2. 如果 NORMAL：CAS CREATED → WAIT_SHIP
3. 如果 GROUP_BUY：CAS CREATED → WAIT_GROUP
4. CAS 成功：
   - NORMAL：发送 ConfirmInventoryEvent、CreateFulfillmentEvent
   - GROUP_BUY：发送 GroupMemberPaidEvent
5. CAS 失败：说明订单已取消/关闭，发送 RefundPaymentCommand 给支付服务
```

普通订单支付成功后直接待发货。

拼团订单支付成功后等待成团，不立即履约。

### 10.5 成团成功消费

拼团服务发送 GroupTeamSuccessEvent，订单服务消费。

流程：

```text
1. 查询 team 下所有 GROUP_BUY 订单
2. 对每个订单 CAS WAIT_GROUP → WAIT_SHIP
3. CAS 成功后发送 ConfirmInventoryEvent
4. CAS 成功后发送 CreateFulfillmentEvent
```

### 10.6 超时关单

订单创建时发送 15 分钟延时消息。

消费流程：

```text
1. 查询订单
2. 如果 status=CREATED，则 CAS CREATED → CANCELLED
3. CAS 成功后发送 ReleaseInventoryEvent
4. 如果是拼团订单，发送 CancelGroupMemberEvent
5. CAS 失败说明支付或其他流程已抢先处理，直接结束
```

### 10.7 用户主动取消订单

只允许取消自己的待支付订单。

流程与超时关单一致：

```text
校验用户身份 → CAS CREATED → CANCELLED → 释放库存 → 取消团成员
```

### 11.3 createPayment

支付接口有两个入口场景：

- 下单成功后前端立即支付。
- 用户在订单页点击立即支付。

流程：

```text
1. 根据 orderId 调订单服务查询订单金额、状态、标题
2. 只允许 CREATED 状态创建支付
3. 插入 t_payment_order，status=INIT
4. 如果插入失败：

   - INIT：可重新调用支付宝，因为支付宝 out_trade_no 幂等
   - SUCCESS：返回已支付
5. 调支付宝 pageExecute 生成支付表单/链接
6. 本地事务更新 pay_url，状态 INIT → PAYING
7. 返回 payUrl/formHtml
```

注意：调用支付宝不放在数据库事务中，避免长事务占用连接。

### 11.4 支付宝回调 notify

```text
out_trade_no      我方 orderId/paymentNo，按实现约定

trade_status      TRADE_SUCCESS
Total_amount
app_id
seller_id
sign/sign_type
```

```text

5. 发送 PaymentSuccessEvent 给订单服务
6. 返回 success 给支付宝
```

如果 CAS 失败：

- 已 SUCCESS：幂等返回 success。
- 已 CLOSED/REFUNDED：可触发退款流程或记录异常。

### 11.5 refundPayment

```text

3. CAS SUCCESS → REFUNDING
4. 调支付宝 alipay.trade.refund
5. 成功后 REFUNDING → REFUNDED
6. 失败则记录失败原因，等待重试/人工介入
```

```text
2. 调订单服务：订单进入 REFUNDING
3. 调库存服务：refundInventory(refundId)
4. 如果拼团订单，调拼团服务：refundMember(refundId)
5. 调履约服务：refundFulfillment(refundId)
6. 调支付服务：refundPayment(refundId)
7. 调订单服务：订单进入 REFUNDED
8. CAS refund_order REFUNDING → SUCCESS
```

失败处理：

- 每个被调服务都必须以 refundId 做幂等。
- 编排失败后记录失败原因，允许重试。
- 支付退款失败通常需要进入人工处理。

## 14. 逆向流程统一设计

### 14.1 超时关单

```text
OrderCloseDelayEvent
  → 订单服务
    → CAS CREATED → CANCELLED
    → ReleaseInventoryEvent
    → CancelGroupMemberEvent（拼团）
```

### 14.2 用户主动取消订单

```text
用户取消
  → 订单服务校验用户身份
  → CAS CREATED → CANCELLED
  → ReleaseInventoryEvent
  → CancelGroupMemberEvent（拼团）
```

### 14.3 超时关团

```text
TeamCloseDelayEvent
  → 拼团服务
    → CAS FORMING/WAIT_LEADER_ORDER → FAILED_TIMEOUT
    → 查询未终态成员
    → TeamTimeoutCancelOrderEvent
      → 订单服务取消订单/发起退款
```

如果成员未支付：走取消订单释放库存。

如果成员已支付但未成团：订单服务需要触发退款，并释放/回滚相关库存状态。

### 14.4 支付成功与关单冲突

```text
支付成功：CREATED → WAIT_GROUP/WAIT_SHIP
超时关单：CREATED → CANCELLED
```

谁先 CAS 成功谁继续。失败方停止或补偿：

- 支付成功方抢失败：通知支付服务退款。
- 关单方抢失败：说明支付已进入后续流程，直接结束。

## 16. 幂等、并发与补偿总原则

### 16.1 幂等键选择

```text
下单主链路：requestId
订单创建：requestId
库存锁定：requestId
拼团占位：requestId
支付创建：orderId + channel 或 paymentNo
支付回调：orderId/paymentNo + channel_order_no
库存确认：orderId + CONFIRM
库存释放：orderId + RELEASE
库存退回：refundId + REFUND
拼团支付确认：teamMemberId + MEMBER_PAID
订单取消：orderId + CANCEL
退单编排：refundId
MQ 消费：messageId + consumerGroup
```

### 16.2 并发控制方式

1. 唯一键插入：适用于幂等表、流水表、订单 requestId。
2. CAS 状态更新：适用于订单、支付、团、团成员状态抢占。
3. 条件库存更新：适用于库存扣减。
4. 短 TTL 分布式锁：只用于入口削峰，不作为正确性唯一保证。
5. 消息消费幂等表：防 MQ 重复投递。

### 16.3 幽灵扫描任务

必须设计定时扫描：

库存幽灵扣减：

```text
t_inventory_deduction status=SUCCESS 但长时间找不到 orderId 或订单未创建成功
→ 调 compensateInventory
```

拼团幽灵占位：

```text
team_member status=PENDING_ORDER 长时间未绑定 orderId
→ cancelMember
```

订单 Outbox：

```text
t_order_outbox status=INIT/FAILED
→ 重发消息
```

支付 INIT 卡住：

```text
payment_order status=INIT 长时间未转 PAYING
→ 关闭或允许重新生成支付链接
```

退款中卡住：

```text
refund_order/payment_order status=REFUNDING 长时间未完成
→ 重试或人工介入
```

## 20. 待和你对齐的问题

以下问题需要进一步确认，确认后可继续更新本文档：

1. 拼团订单库存确认时机：是否采用本文建议的“订单进入 WAIT_SHIP 时统一 confirmInventory”？即普通订单支付成功确认库存，拼团订单成团成功确认库存。
2. 拼团失败后已支付成员的处理：是否统一由订单服务触发退款，并释放锁定库存？
3. 同一用户同一活动是否只能参与一次？如果用户退款后是否允许再次参与同一活动？
4. 团成功后成员退款，团状态是否永远不回滚 SUCCESS？本文按“不回滚”设计。
5. 活动限额退单后是否退回 available_quota？本文按“退回”设计，但实际运营上也可以选择不退回。
6. 普通订单是否也使用 15 分钟支付超时？本文按普通和拼团统一使用延时关单设计。
7. 支付宝 out_trade_no 使用 orderId 还是 paymentNo？本文倾向 paymentNo 更符合支付流水语义，但草稿里用 orderId，需要最终统一。
8. 退单是否允许部分退款/部分退货？本文按整单退设计。
9. 一个订单是否只包含一个 SKU？当前文档按单 SKU 下单建模。如果未来支持购物车多 SKU，需要引入 OrderItem 与多库存锁定事务。
10. 履约服务是否需要反向通知订单服务发货/收货？本文按需要通知设计。
11. 拼团 Top N 排序是否只按“还差支付人数”，是否需要加入 expireTime、创建时间、热度等二级排序？
12. 库存服务的幽灵扫描如何判定“订单未创建成功”：是同步查订单服务，还是订单创建成功后回写 orderId 到 t_inventory_deduction？本文建议两者都可，优先订单成功后回写 orderId。
13. MQ 使用 RocketMQ 事务消息还是本地 Outbox + 定时投递？本文主要按 Outbox/事务消息理念描述，落地时需要统一具体实现方式。
14. 退单服务编排采用同步 Dubbo 逐步调用，还是每一步都用 MQ 命令事件？本文按退单服务同步/可重试编排描述。
15. 是否需要独立活动服务？当前文档把活动读取部分仍放在商品服务/拼团服务上下文中，未单独拆活动服务。

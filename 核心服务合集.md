## 下单链路

下单接口在订单服务里：**orderForOrderService**

**活动下单和普通下单的路径不同，我们orderForOrderService接口得根据前端传过来的标志位判断是活动还是普通。**





### 1. 订单与下单接口的幂等

用户第一次点击下单，前端发现内存里没有requestId，就会调用组件生成一个**UUID格式的requestId**，然后放在内存里。

一个requestId就是**一个下单请求的唯一标志**，requestId会存放在前端的内存里，用户重复点击、前端超时重试，携带的都是同样的requestId。

此外我们也会利用前端的行为让这次下单时用户无法再进行别的操作，最后当商品详细页面收到上次的结果后才**把原requestId删除**，或者说直接**刷新页面**。

下面会用requestId加个**短TTL的分布式锁**，这个分布式锁的目的是为了**防止用户快速点击两次下单**，同时到达两个下单请求，我们用分布式锁防出去一个，虽然前端会置灰用前端的逻辑避免，但是还是有可能失效的，我们要做双重防护。

之所以**TTL短**是因为我们不想让这个**分布式锁把前端超时重试发的请求拦截**。

其次短TTL分布式锁其实能防住绝大多数情况下的并发，但这并不稳健，因此我们下单链路中每个节点都还是需要**防止并发**的机制的。



我们并没有引入在接口前面查一次表的操作，因为没啥意义，加了还多走一次数据库查询，使得性能下降。

查一次表服务场景是：前面的请求已经做好了，只是响应没发回去，我们加一个查表可以立马返回。

但是这种场景很少，属于为了优化1%的情况，使得其他99%的情况性能下降了，整个接口的平均性能暴降，没意义。

```plsql
1. 前一个完成了只是没响应：我们希望重试请求直接查表把orderId返回来。
先加锁再查表就多了一次加锁RT，不如先查表的方式。

2. 这个请求是个紧跟进来的第二次点击下单溜进来的。
如果是这样的话我们的方式不如先加锁，先加锁直接就拦住了
我们不会搞什么没抢到锁sleep什么的，没必要啊，让前端再发一个重试呗就。
3. 前一个在某个地方拉了，重试进来了
这种情况先加锁还是先查表一样了。
```

### 2. 全链路的幂等避免同步写操作重试带来的弊端

我们通过短TTL分布式锁 + 按钮置灰分别从**后端和前端**解决了**用户重复下单**的问题，在下单的时候我们也利用**前端的行为**让用户没法儿在当前商品详细页面上进行其他的操作。

后面有可能请求超时，前端会重试，涉及到重试就有几种场景：

```java
场景一：请求 A 扣完库存后崩了，请求 B 进来
B → 扣库存：查 stock_deduction_log，requestId 存在 → 跳过，视为已扣
B → 拼团：没做过 → 正常做
B → 创建订单：没做过 → 正常做
没问题，B 自动从断点接上了。

场景二：请求 A 拼团成功后崩了，请求 B 进来
B → 扣库存：幂等跳过
B → 拼团：查 group_buy_team_member，已有记录 → 天然幂等跳过
B → 创建订单：正常做
也没问题。

场景三：请求 A 全做完了但响应丢了，请求 B 重试

B → 扣库存：幂等跳过
B → 拼团：幂等跳过
B → 创建订单：requestId 已存在 → 查出 orderId 直接返回
```

我们会通过内部每个节点的幂等机制，解决dubbo同步写操作重试带来的问题。

### 5. dubbo同步写库存服务

调用库存服务的下单接口：**lockInventory**

1. 库存扣减幂等表

我们这里会有一张专门以**requestId为幂等key**的扣库存幂等表来标识此**下单请求**之前有没有扣过库存，因为扣库存操作本身不会留下业务实体，我们得单独设置一个幂等表，幂等key为requestId也是由于requestId他就是我们设置的**一次下单请求**的唯一标识，他的语义使得它在全链路都可以作为这个幂等key。



除此之外这个表必须要有一个status状态位。

有人就要问了，你这个操作就是一个大的事务方法，有幂等表就说明做成功了，为什么还要这个状态位？
原因是我们这个**lockInventory接口是被同步调用的，这里有幽灵扣减的风险（最后订单因为各种原因没创建，但是库存扣了）。**

**这种情况我们必须给一个定时任务扫库存扣减幂等表，发现这种之后给加回来。**

**没有状态位的话我们不知道这个操作成没成，有可能又重复补偿了。**

**有人可能说你大可以把那条记录给删了来表明补偿做了啊**

**那我们实践里delete是一个危险的操作，我们希望数据库的东西留下了就是留下了，别删除**

```plsql
CREATE TABLE `t_inventory_deduction` (

    `request_id` VARCHAR(64) NOT NULL COMMENT '下单请求ID，幂等键',
    `sku_id` BIGINT NOT NULL,
    `quantity` INT NOT NULL COMMENT '扣减数量',
    `status` VARCHAR(16) NOT NULL COMMENT 'PROCESSING/SUCCESS/FAILED/CANCELED',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_request_id` (`request_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='库存扣减幂等表-仅下单链路';
```

1. **库存流水表t_stock_log**

我们刚才那个库存扣减幂等表专门给下单这个流程服务的，定时任务扫这个表来得知有没有幽灵扣减。

但是我们还有另外一些库存操作：支付、超时/取消、退款。

1. 防止并发

虽然我们通过短TTL分布式锁防止了用户多次下单并且避免了大多数情况下的并发。

但还是有可能并发的，并发的话我们就得靠每个节点来防止了。

我们这里的编排是在扣库存接口的流程就是一个大事务方法

- **插入库存扣减幂等表**

三种情况：

- 插入成功，说明它是第一个到的，让它推进。
- 前一个事务没提交重试请求来了，他会在insert这里阻塞。
- 前一个事务提交了，那insert这里直接就插入失败，然后我们直接给响应了。
- 前一个事务回滚了，这里第二个请求就插入了。

1. redis删库存缓存

我们用redis存库存其实目的就是方便读取，没有任何预校验的意味。

和spu、sku一样，用cache aside方式，更新就删。

不在redis里加减，更新了直接删，有读的去数据库重新拉取。

这个方案相比我们在redis里加减的优劣：


### 10. 性能分析

虽然说我们已经在下单接口的起点那里通过短TTL分布式锁尽可能避免了并发出现，但还是有这种可能，比如库存节点遇到并发不会有一致性问题，但是可能有一点点性能问题，比如第一个请求进来在库存节点那里还没做完，第二个请求进来insert阻塞，这个阻塞就占用数据库连接，并且使得我们的订单服务线程还得等待。

但是claude说这个实际影响其实不大：

1. 概率极低：短TTL分布式锁把99%的并发挡在外面了，能穿透到节点内部的并发只有锁刚好过期那一瞬间的情况。
2. 阻塞时间很短：每个节点的事务就是几个 SQL 操作，正常情况下几十毫秒内就 commit 了。第二个线程等的时间也就是这个量级。
3. 这是正确的等待：第二个线程必须知道第一个的结果才能决定自己的行为（跳过还是接管），这个等待是语义上必要的。**为什么我们愿意在这种情况下等待？因为重试线程需要知道自己接下来的行为，他不是说碰到冲突了就跑了，它的思路是碰到冲突我等最后结果，你没做成，我来做，你做成了，那我下一步。**

支付接口最前端用一个insert交易流水表（requestId）来防止并发和幂等

流水表初始状态为init。

然后调用支付宝接口，传递各种参数，最后得到payurl。

再开一个事务写进交易流水表里，然后状态从init变成paying。

**而且有可能支付方式有变更，变更的话我们希望只在支付服务范围内。**



### 对接支付宝

我们目前只支持对接支付宝web支付

一、前期准备

1. **注册账号并签约**：首先，需要注册支付宝商家平台账号并完成企业认证，然后在产品中心“签约”【电脑网站支付】产品。
2. **创建应用并获取密钥**：在支付宝开放平台创建应用，使用官方推荐的**RSA2签名算法**生成一对密钥。
3. **获取关键参数**：在应用中获取以下参数，它们是后续API调用的“身份证”：

- **APPID**：应用的唯一标识。
- **应用私钥**：自己持有的密钥，用于生成签名。
- **支付宝公钥**：支付宝的密钥，用于验证来自支付宝的通知真伪。

1. **配置回调地址**：在开发环境或生产应用中，分别设置好用于接收支付结果的**异步通知地址（notify_url）** 和**同步跳转地址（return_url）**，异步通知地址就是支付服务的结算接口的地址，同步跳转地址就是支付成功后支付宝帮你跳转到目标前端页面。

二、核心发起

后端调用支付宝SDK，将订单信息发送给支付宝，并获取一个能引导用户跳转的支付链接。

三、校验

我们和支付宝有一对非对称密钥，我们有两个地方需要校验：

1. 在调支付宝生成支付单时，我们会用私钥根据请求参数搞一个数字签名，支付宝用公钥解密，要是没问题，**说明这个请求没问题，确实是和我签了协议的公司发来的请求，没和我申请过的我不受理。**
2. 在支付宝收到款了调结算接口之前它会用私钥根据参数生成一个签名，结算接口用自己的公钥来解析。



我们是有两对密钥，因为两个操作的发起方和接收方不一样。

第一个电商平台发起的，第二个支付宝发起的。



## 结算接口

支付宝收到款项后调用我们给它的回调接口，这个通知只是纯粹的通知我们电商后台，他收到钱了，没收到电商的回复他就重试，收到了他就走人。

而不会出现一种情况就是后端和它响应“没法儿结算，你退款吧”。

退款是需要支付服务主动调用支付宝退款接口去做的。



结算天然会和超时关单和取消订单冲突的，



用户支付前或者支付时直接手动取消订单，取消订单和超时关单要做的事儿基本一致，都是下单的逆向操作。

结算和超时关单与取消订单的流程很相似，都是先改订单状态，然后处理库存和拼团

我们必须保证这两个东西不并发，**因此谁先抢先把订单状态流转了，谁先做**。





支付宝的回调是打进支付服务的，和第三方支付通信的只能是支付服务，

支付服务根据orderId改流水表的状态，支付流水表只是表达一个单支付与否以及第三方的支付信息是什么。由此就ok了

成功改了之后利用**rocketmq异步通知订单服务**，订单服务来抢状态，抢到了异步通知库存和拼团。



如果订单的结算抢状态失败了，直接通知支付服务，调支付服务的退单接口，和第三方支付能力提供方通信，处理退钱。







我们的结算的主要动作就是改

支付流水状态、库存操作、订单操作、拼团操作。

结算的特性使得后面三个操作可以异步做，支付的结果并不靠这三个操作，用户付完钱就是支付完成了，其实就连调结算接口这个操作也并不实时，甚至可以说是异步的。

同步的目的就是快，但是哪怕你是同步，也有空隙，用户付完款直接去全部订单找订单，发现订单还是待支付，因为结算接口还没调或者还没执行完。

和异步的效果差不了多少，只是这个窗口期长短的事儿，用rocketmq的事务方法还更方便。

### 一、支付服务的结算接口

结算接口是我们支付接口给支付宝传递的notify_url，然后在用户支付后支付宝来调用的。

这个接口的形参取决于支付宝传什么过来：

```java

2. trade_no 支付宝给这个订单的交易流水号
3. trade_status 对于电脑网站支付，当交易成功时，你会收到的状态是 TRADE_SUCCESS

6. seller_id：收款方的支付宝账号，确保收款方正确
```

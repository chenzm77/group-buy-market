# 商品服务架构第二版

## 1. 服务定位

商品服务负责拼团电商中的商品展示、商品搜索、商品详情和活动售卖配置。

当前商品服务同时承载两个强相关上下文：

1. **商品上下文**：SPU、SKU、商品上下架、商品详情、价格、销量、标签。
2. **活动上下文**：活动、活动 SKU、活动价、限购、活动状态。

活动和商品强绑定，目前没有拆分活动服务的计划。活动长期放在商品服务内部实现，避免为了强关联业务引入额外 RPC 调用和一致性复杂度。

商品服务不负责真实库存。真实库存、库存校验、库存扣减、库存回滚都归库存服务。商品详情如果需要展示库存，可以调用库存服务查询，商品服务不在 SKU 中维护库存冗余。

---

## 2. 已确定架构决策

### 2.1 活动不拆服务

活动长期放在商品服务。

原因：

- 活动和 SPU/SKU 强绑定。
- 商品详情、商品搜索、活动列表都需要聚合商品和活动信息。
- 当前阶段拆活动服务会增加系统复杂度，收益不明显。

### 2.2 不做类目树

商品不维护传统多级类目树，只保留扁平标签 `tags`。

原因：

- 当前拼团电商更重搜索、推荐和活动商品展示。
- 类目入口可以转化为标签或关键词搜索。
- 扁平标签更适合初版快速落地。

### 2.3 商品详情使用结构化 JSON

`t_spu.description` 存结构化 JSON。

不存 HTML，也不只存对象存储 URL。

结构化 JSON 用来表达详情页模块，例如图片、文本、参数、说明等。前端根据结构化 JSON 渲染详情页。

### 2.4 ES 不保存 description

ES 文档不保留 `description`，也不基于 `description` 搜索。

初版只搜索：

- `spuName`
- `tags`

原因：

- 商品详情描述较长，噪声多，容易影响搜索相关性。
- 商品名称和标签已经能覆盖初版搜索需求。
- ES 文档只保存搜索和列表展示需要的最小字段。

### 2.5 activityStatus 只表示有没有活动

`activityStatus` 只表示是否存在当前有效活动。

取值：

- `0`：无当前有效活动。
- `1`：有当前有效活动。

不区分未开始、进行中、已结束。未开始和已结束都不算当前有效活动。

### 2.6 商品详情缓存使用 Cache-Aside

商品详情缓存使用 Cache-Aside，不采用逻辑过期。

原因：

- 商品详情是读多写少场景。
- 热点商品希望尽量常驻缓存，通常不设置短 TTL。
- Cache-Aside 下，更新时主动删除缓存，缓存缺失是正常情况。
- 逻辑过期主要解决热点 key 自然过期导致的阻塞问题，但当前热点 key 不依赖自然过期。

基础流程：

```text
更新 MySQL
  -> 删除 Redis
    -> 广播删除本地 Caffeine
```

缓存未命中时，可以用互斥锁或短暂自旋保护 MySQL。

### 2.7 初版不引入 Canal

ES 同步初版不引入 Canal。

原因：

- Canal 会增加部署、维护和理解成本。
- 会影响 SPU ES 文档的业务操作范围明确。
- 可以在业务操作成功后主动发送索引重建消息。
- 消费者收到消息后重新计算一次 SPU document 并覆盖写 ES。

同步链路：

```text
业务写操作成功
  -> 发送 SpuIndexRebuildMessage(spuId)
    -> 商品服务索引消费者
      -> 查询 MySQL 聚合数据
        -> 重建 spu_index 文档
          -> 写入 ES
```

### 2.8 商品服务不维护库存冗余

商品服务不在 SKU 或 ActivityItem 中维护库存冗余。

原因：

- 库存不是商品服务的核心事实数据。
- 真实库存由库存服务负责。
- 冗余库存会带来同步问题，但对当前详情查询收益有限。
- 商品详情本身已经需要聚合多个数据源，库存可以直接查库存服务。

## 3. 核心职责

### 3.1 商品管理

- 创建、修改、上架、下架 SPU。
- 创建、修改、启用、禁用 SKU。
- 维护商品主图、轮播图、结构化详情、扁平标签、销量展示值。
- 维护 SKU 规格、价格、图片。

### 3.2 活动管理

- 创建、启用、禁用活动。
- 配置活动关联的 SPU。
- 配置活动 SKU、活动价、单用户限购。
- 判断活动是否当前有效。

### 3.3 商品查询

- 商品详情查询：根据 SPU 查询商品详情、SKU 列表、当前有效活动。
- 商品搜索：关键词搜索、标签匹配、活动过滤、销量排序、价格排序、上新排序。
- 首页商品列表：优先展示正在活动中的商品。

### 3.4 搜索索引维护

- MySQL 是商品事实数据源。
- Redis/Caffeine 是商品详情查询缓存。
- Elasticsearch 是商品搜索和列表展示查询模型。
- 业务写操作成功后发送 SPU 索引重建消息。

---

## 4. 领域边界与聚合

商品服务初版包含两个聚合：

1. 商品聚合：`Spu` 作为聚合根，`Sku` 作为实体。
2. 活动聚合：`Activity` 作为聚合根，`ActivityItem` 作为实体。

活动虽然不拆服务，但领域模型上仍然和商品聚合分开，避免 `Spu` 聚合承担过多职责。

---

## 5. 商品聚合

### 5.1 聚合根：Spu

SPU 表示一个可被用户认知的商品，例如“iPhone 16”。

核心字段：

- `id`
- `name`
- `description`
- `mainImage`
- `images`
- `tags`
- `sales`
- `status`
- `createTime`
- `updateTime`

字段说明：

- `description` 是结构化 JSON。
- `tags` 是扁平标签。
- `sales` 是展示销量，不要求强实时。
- `status` 表示草稿、上架、下架。

核心行为：

- 创建商品。
- 修改基础信息。
- 修改结构化详情。
- 修改图片。
- 修改标签。
- 上架。
- 下架。
- 更新展示销量。

### 5.2 实体：Sku

SKU 表示 SPU 下的具体规格组合，例如“红色 + 128G”。

核心字段：

- `id`
- `spuId`
- `specJson`
- `price`
- `image`
- `status`
- `createTime`
- `updateTime`

字段说明：

- `price` 单位分。
- `specJson` 存规格组合，例如 `{"颜色":"红色","容量":"128G"}`。
- SKU 不维护库存冗余。

核心行为：

- 创建 SKU。
- 修改规格。
- 修改价格。
- 修改图片。
- 启用。
- 禁用。

### 5.3 值对象候选

- `Money`：价格，单位分。
- `Spec`：规格组合。
- `ProductImage`：商品图片 URL。
- `ProductTag`：商品标签。
- `ProductDescription`：结构化详情 JSON。

## 6. 活动聚合

### 6.1 聚合根：Activity

活动表示某个 SPU 的一次拼团/秒杀售卖配置。

核心字段：

- `id`
- `spuId`
- `name`
- `startTime`
- `endTime`
- `groupSize`
- `status`
- `createTime`

核心行为：

- 创建活动。
- 启用活动。
- 禁用活动。
- 判断活动是否当前有效。
- 修改活动时间。
- 修改成团人数。

### 6.2 实体：ActivityItem

活动 SKU 配置，表示某个 SKU 在活动中的价格和限购规则。

核心字段：

- `id`
- `activityId`
- `skuId`
- `seckillPrice`
- `perUserLimit`
- `createTime`

字段说明：

- `seckillPrice` 是活动价，单位分。
- `perUserLimit` 是单用户限购数量。
- ActivityItem 不维护活动库存冗余。

核心行为：

- 修改活动价。
- 修改单用户限购。

SPU 主表。

核心字段：

- `tags`

说明：

### 7.2 t_sku

SKU 表。

核心字段：

- `id`
- `spu_id`
- `spec_json`
- `price`
- `image`
- `status`

说明：

- 不维护 `stock` 作为核心字段。

## 8. 商品详情缓存设计

### 8.1 缓存层级

```text
Caffeine 本地缓存
  -> Redis
    -> MySQL
```

查询流程：

1. 查 Caffeine。
2. 本地未命中，查 Redis。
3. Redis 未命中，查 MySQL。
4. MySQL 查询成功后，写 Redis，再写 Caffeine。

### 8.2 Redis Key

```text
spu:{spuId}
sku:{skuId}
spu:sales:{spuId}
```

说明：

- `spu:{spuId}` 保存 SPU 基础详情 JSON。
- `sku:{skuId}` 保存 SKU 基础详情 JSON。
- `spu:sales:{spuId}` 单独保存销量展示值。
- 不设计 `sku:stock:{skuId}` 作为商品服务核心缓存。

### 8.3 Cache-Aside 一致性

商品基础信息更新时：

1. 更新 MySQL。
2. 删除 Redis 缓存。
3. 发送缓存失效广播消息。
4. 所有商品服务实例收到消息后删除本地 Caffeine 缓存。
5. 发送 SPU 索引重建消息。

### 8.4 缓存问题处理

缓存穿透：

- 对不存在的 SPU/SKU 缓存空值。
- 空值 TTL 设短，例如 2 分钟。

缓存击穿：

- Redis 未命中时使用互斥锁保护 MySQL。
- 抢到锁的请求查询 DB 并回填缓存。
- 没抢到锁的请求短暂 sleep 后重查 Redis。

缓存雪崩：

- 热点商品基础数据可以长期驻留。
- 非热点数据可以设置较长 TTL 并增加随机抖动。

## 9. Elasticsearch 搜索设计

ES 只负责搜索和列表页展示，不作为商品事实数据源。

搜索场景包括：

- 关键词搜索。
- 标签匹配。
- 活动商品过滤。
- 销量排序。
- 价格排序。
- 上新排序。
- 首页活动商品推荐。

初版只搜索：

- `spuName`
- `tags`

不搜索 `description`。

不做类目树，SPU 使用扁平 `tags`，搜索统一走 ES。

## 10. ES 文档模型

ES 中以 SPU 为文档粒度，一个 SPU 一个 document。

文档由以下数据聚合而成：

- `t_spu`
- `t_sku`
- `t_activity`
- `t_activity_item`

### 10.1 spu_index 文档

```json
{
  "spuId": 10001,
  "spuName": "iPhone 16",
  "tags": ["5G", "旗舰", "iOS", "面部识别"],
  "mainImage": "https://xxx.jpg",
  "minPrice": 599900,
  "activityMinPrice": 500000,
  "sales": 158000,
  "activityStatus": 1,
  "createTime": "2026-05-01T00:00:00Z"
}
```

字段说明：

- `spuId`：SPU ID。
- `spuName`：商品名称，用于全文检索。
- `tags`：商品标签，用于全文检索。
- `mainImage`：列表页主图，不参与索引。
- `minPrice`：SKU 最低普通价。
- `activityMinPrice`：当前有效活动 SKU 的最低活动价。
- `sales`：展示销量。
- `activityStatus`：是否存在当前有效活动。
- `createTime`：创建时间，用于上新排序。

### 10.2 ES Mapping 核心字段

```json
{
  "mappings": {
    "properties": {
      "spuId": { "type": "long" },
      "spuName": {
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"
      },
      "tags": {
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_smart"
      },
      "mainImage": { "type": "keyword", "index": false },
      "minPrice": { "type": "long" },
      "activityMinPrice": { "type": "long" },
      "sales": { "type": "long" },
      "activityStatus": { "type": "byte" },
      "createTime": { "type": "date" }
    }
  }
}
```

## 11. ES 查询接口设计

### 11.1 请求参数

```java
public class ProductSearchRequest {
    private String keyword;
    private String sort;
    private Integer page;
    private Boolean activityOnly;
}
```

字段说明：

- `keyword`：搜索关键词。为空表示列表查询。
- `sort`：`default`、`sales`、`price_asc`、`price_desc`、`newest`。
- `page`：页码，从 1 开始。
- `activityOnly`：是否只看活动商品。

每页大小初版固定 20。

### 11.2 查询规则

- `keyword` 不为空时，使用 `multi_match` 查询 `spuName^3` 和 `tags^2`。
- `activityOnly=true` 时过滤 `activityStatus=1`。
- `default` 按 `_score` 降序。
- `sales` 按 `sales` 降序。
- `price_asc` 按 `minPrice` 升序。
- `price_desc` 按 `minPrice` 降序。
- `newest` 按 `createTime` 降序。

### 11.3 分页限制

初版使用 `from + size` 分页。

限制：

- 每页固定 20。
- 最大页数限制为 100 页。

后续如果要做 Feed 流式浏览，再考虑 `search_after`。

## 12. ES 索引同步设计

### 12.1 同步原则

ES 文档是查询模型，不是事实数据源。

只要影响列表展示或搜索结果的业务操作成功，就发送 SPU 索引重建消息。

初版不引入 Canal。

### 12.2 触发索引重建的业务操作

以下操作成功后，需要发送 `SpuIndexRebuildMessage(spuId)`：

1. SPU 信息增删改。
2. SPU 销量更新。
3. SKU 价格更新。
4. Activity 创建、启用、禁用。
5. Activity 过期。
6. ActivityItem 创建。
7. ActivityItem 价格更新。

### 12.3 活动过期处理

活动过期不会天然触发业务写操作，因此需要额外机制触发索引重建。

可选方案：

- 定时任务扫描过期活动并发送索引重建消息。
- 创建活动时发送延迟消息，到期后触发索引重建。

初版可以先用定时任务，后续再优化为延迟消息。

### 12.4 消息设计

```java
public class SpuIndexRebuildMessage {
    private Long spuId;
    private String reason;
    private Long eventTime;
}
```

`reason` 可选值：

- `SPU_CHANGED`
- `SPU_DELETED`
- `SALES_CHANGED`
- `SKU_PRICE_CHANGED`
- `ACTIVITY_CHANGED`
- `ACTIVITY_EXPIRED`
- `ACTIVITY_ITEM_CHANGED`

### 12.5 重建流程

输入：`spuId`

流程：

1. 查询 SPU。
2. 如果 SPU 不存在或不可搜索，删除 ES 文档。
3. 查询该 SPU 下可用 SKU。
4. 计算 `minPrice`。
5. 查询该 SPU 当前有效活动。
6. 查询活动 SKU 配置。
7. 计算 `activityMinPrice`。
8. 判断 `activityStatus`。
9. 组装 `SpuIndexDoc`。
10. 覆盖写 ES。

### 12.6 幂等与可靠性

索引重建消息需要支持重复消费。

重建逻辑从 MySQL 读取最新事实数据后覆盖写 ES，天然适合幂等。

初版可以接受重复重建。如果后续消息量过大，再考虑按 `spuId` 合并、延迟重建或去重。

## 13. 商品详情查询设计

商品详情页不直接走 ES。

原因：

- ES 文档只保存列表展示字段。
- 详情页需要完整 SPU、SKU、结构化详情、活动配置。
- 详情页读多写少，适合多级缓存。

详情查询流程：

```text
queryProductDetail(spuId)
  -> 查 Caffeine
  -> 查 Redis
  -> 查 MySQL 聚合 SPU/SKU/Activity/ActivityItem
  -> 需要库存时查询库存服务
  -> 回填 Redis
  -> 回填 Caffeine
  -> 返回详情 DTO
```

详情返回内容初版包括：

- SPU 基础信息。
- 结构化图文详情 JSON。
- 主图和轮播图。
- 标签。
- SKU 列表。
- 当前有效活动。
- 活动 SKU 价格和限购。
- 展示销量。
- 库存服务返回的库存信息，如果该接口需要展示库存。

## 14. 与其他服务的关系

### 14.1 库存服务

商品服务不负责真实库存扣减，也不维护 SKU 库存冗余。

真实库存扣减、库存校验、库存回滚由库存服务负责。

商品详情需要库存时，可以通过库存服务查询。

### 14.2 订单/支付服务

支付成功后，订单或支付服务发送消息给商品服务，异步更新 SPU 展示销量。

商品服务更新销量后，发送 SPU 索引重建消息，最终更新 ES 中的 `sales`。

商品销量不要求强实时。

### 14.3 网关/前端

前端商品搜索、首页列表、活动商品列表走商品搜索接口。

商品详情页走商品详情接口。

### 15.3 app

放用例编排：

- 商品创建/修改/上下架。
- 商品详情查询。
- 搜索查询编排。
- 活动创建/修改/上下线。
- 索引重建用例。
- 发送索引重建消息。

### 15.4 trigger

放外部入口：

- Dubbo RPC。
- HTTP Controller，如果后续需要。
- MQ Consumer，如果按架构习惯也可放 trigger，再调用 app。

## 16. 当前版本结论

商品服务当前架构核心结论：

1. 活动长期放在商品服务内，不拆活动服务。
2. 商品不做类目树，只做扁平标签。
3. 商品详情 `description` 存结构化 JSON。
4. ES 不保存也不搜索 `description`，只搜索 `spuName + tags`。
5. `activityStatus` 只表示是否存在当前有效活动。
6. 商品详情缓存使用 Cache-Aside，不采用逻辑过期。
7. 初版不引入 Canal，改用业务事件 + MQ 触发 SPU 索引重建。
8. 商品服务不维护 SKU 库存冗余，库存归库存服务。

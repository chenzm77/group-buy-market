多级缓存
商品服务是典型的读多写少场景，一个SPU详情页可能每秒几千次请求，但商品信息可能几天才改一次。这是做多级缓存的最佳场景。
方案：本地缓存Caffeine + Redis + MySQL
SPU、SKU detail的查询链路：
● 先查Caffeine本地缓存（微秒级）
● 本地没有 → 查Redis（毫秒级）
● Redis没有 → 查MySQL → 写Redis → 写Caffeine
写入链路（商品更新时）：
● 更新MySQL
● 删除Redis缓存
● 发一条RocketMQ广播消息
● 所有商品服务实例收到广播 → 清除本地Caffeine缓存
几个问题：
1. "本地缓存和Redis不一致怎么办？" → 广播消息清除本地缓存，存在秒级延迟，商品信息不是强一致场景，可接受
2. "缓存击穿怎么防？" → 用互斥锁（Redis setnx），只有一个请求穿透到DB，其他请求等锁或返回旧值
3. "缓存穿透怎么防？" → 缓存空值，TTL设短一点（比如2分钟）
4. "缓存雪崩怎么防？" spu和sku都久居
   库存与销量问题
   spu的销量和sku的库存我们会在redis里存放的
   查询spu和sku详细时我们会查redis里的这两类字段
   因为商品详细页面不需要精准的库存和销量数字
   比如销量我们只需要知道大概，比如100+
   库存我们更是只需要知道有没有，有的话大概有多少。
   spu销量由支付服务发事务消息异步更新，消费者是商品服务的
   sku库存和activity_item活动余额因为是库存服务负责的，直接库存服务就扣了。
   互斥锁避免缓存击穿
   抢到锁的自然就是去查DB，然后写redis，写caffeine。
   redisson锁的trylock支持waitTime 参数（比如 100ms），线程会阻塞等待最多 100ms。如果 100ms 内锁被释放，它会获得锁并继续执行；如果超时则返回 false。
   用redisson锁会导致大量线程阻塞，大量线程阻塞代价太大了，所以这种情况下我们还是希望快速失败 + 自旋。
   没抢到锁sleep然后继续查redis，没有继续sleep
   最多尝试50次。
   上Canal
   Canal 本质上就是一个伪装成 MySQL 从节点的 binlog 订阅者。它的核心能力就一个：把 binlog 解析成结构化的变更事件。
   Canal的Server只有感知能力和投递能力，没有处理能力
   Canal有两种架构
1. Canal Server + Canal Client（原生模式）
   MySQL → Canal Server → Canal Client（你的 Java 应用）→ 自己处理
   写一个Canal Client和Server对接，进行各种处理。
   太麻烦
2. Canal + MQ（主流模式）
   Canal 直接把变更事件投递到 MQ，下游消费者各取所需。这是生产环境的主流用法。

   因为ES中我们的spuIndex存放的东西不只是sup本身，其实是spu、sku、activity、activity_item的聚合，而不是不是一个简单的聚合，每个表只有一些东西是spu需要的。

   public class ProductSearchRequest {
   private String keyword;        // 搜索词，为空就是全量
   private String sort;           // default / sales / price_asc / price_desc / newest
   private Integer page;          // 页码，从 1 开始
   // 每页固定 20，不需要传
   }

// 响应
public class ProductSearchResult {
private List<ProductItem> items;   // 当前页商品
private Integer total;             // 总数
private Integer totalPages;        // 总页数（最多 20）
private Integer currentPage;       // 当前页
}
public ProductSearchResult search(ProductSearchRequest request) {
/**
ES有专门的查询API，我们直接在接口代码里把查询的DSL拼接，利用ESClient然后传给ES
*/
SearchRequest.Builder builder = new SearchRequest.Builder()
.index("spu_index")
.from((request.getPage() - 1) * 20)
.size(20);

    // 查询条件
    BoolQuery.Builder boolQuery = new BoolQuery.Builder()
        .filter(term(t -> t.field("status").value(1)));

    if (request.getKeyword() != null && !request.getKeyword().isBlank()) {
        boolQuery.must(m -> m.multiMatch(mm -> mm
            .query(request.getKeyword())
            .fields("spuName^3", "tags^2", "description")
        ));
    }

    builder.query(q -> q.bool(boolQuery.build()));

    // 排序
    builder.sort(sortBy(request.getSort()));

    // 执行
    SearchResponse<ProductDoc> response = esClient.search(builder.build(), ProductDoc.class);

    // 映射结果
    return mapToResult(response);
}

private SortOptions sortBy(String sort) {
return switch (sort) {
case "sales"      -> SortOptions.of(s -> s.field(f -> f.field("sales").order(SortOrder.Desc)));
case "price_asc"  -> SortOptions.of(s -> s.field(f -> f.field("minPrice").order(SortOrder.Asc)));
case "price_desc" -> SortOptions.of(s -> s.field(f -> f.field("minPrice").order(SortOrder.Desc)));
case "newest"     -> SortOptions.of(s -> s.field(f -> f.field("createTime").order(SortOrder.Desc)));
default           -> SortOptions.of(s -> s.score(sc -> sc.order(SortOrder.Desc)));
};
}

插入：
{
"spuName": "苹果手机 iPhone 16",
"tags": ["5G", "旗舰"],
}
对每个text字段用ik_max_word 分词
spuName 倒排索引：
苹果  → [doc1]
手机  → [doc1]
iphone → [doc1]
16    → [doc1]

tags 倒排索引：
5g   → [doc1]
旗舰  → [doc1]
一个index，三个字段，三套独立的倒排索引，互不干扰
查询：
用户输入"苹果手机"
→ ik_max_word 分词 → ["苹果", "手机"]
ES拿着分词的结果去三个txt字段的倒排索引里找

一个SPU的index，我们对它的两个字段“spuName、tags进行分词
每个字段都构建了自己的很多倒排索引，比如spuName下面有：苹果、手机、相机
一个对spu的index进行的关键词查询进来了，被分词器分词，分别去各个字段的倒排索引进行匹配，
在spuName下匹配到了苹果和手机两个索引，每个索引下都有一些document，比如苹果下面有个doc1，我们就计算doc1的spuName字段中苹果这个关键词的BM25相关性系数
最后在spuName内部做一个加法，比如doc1不仅在苹果下有，在手机下也有，那doc1在spuName下的BM25相关性得分就是二者的加和。
最后三个字段都搜索完毕，比如doc1在每个字段下都命中了，都有一个BM25相关性得分，最后doc1对这个关键词查询的BM25总得分就是三者的加权和。

一个BM25相关性得分计算的例子：

spuIndex对spuName进行分词，有一个苹果的倒排索引，下面有一个doc1，doc1的spuName为苹果16移动电话12GB + 512GB
我们现在要计算doc1在spuName字段上有关苹果这个关键字的相关性得分：
假设索引里有10000个spu
doc1 的 spuName："苹果16移动电话12GB + 512GB"
分词后：["苹果", "16", "移动", "电话", "12gb", "512gb"]
文档长度 = 6 个词

假设所有 SPU 的  平均长度 = 8 个词
第一步：算IDF(”苹果“)
IDF = ln(1 + (总文档数 - 包含"苹果"的文档数 + 0.5) / (包含"苹果"的文档数 + 0.5))

假设 10000 个 SPU 里，spuName 包含"苹果"的有 50 个：

IDF = ln(1 + (10000 - 50 + 0.5) / (50 + 0.5))
= ln(1 + 9950.5 / 50.5)
= ln(1 + 197.04)
= ln(198.04)
≈ 5.29
越少见越稀有的IDF越高，IDF是全局的
算的是在这个索引的所有文档里的spuName这个字段里，苹果这个词的出现频率。
出现频率越低说明越稀有，命中一次越不容易，命中的这个文档的得分也应该越高
第二步：算TF部分
TF部分 = (f × (k1 + 1)) / (f + k1 × (1 - b + b × 文档长度/平均长度))
= (1 × (1.2 + 1)) / (1 + 1.2 × (1 - 0.75 + 0.75 × 6/8))
= 2.2 / (1 + 1.2 × (0.25 + 0.75 × 0.75))
= 2.2 / (1 + 1.2 × (0.25 + 0.5625))
= 2.2 / (1 + 1.2 × 0.8125)
= 2.2 / (1 + 0.975)
= 2.2 / 1.975
≈ 1.11
TF算的是局部的，你苹果在spuName里出现，出现的频率是多少啊？
出现频率越高，你这个doc1的得分也就相应越高。

第三步：计算BM25("苹果", doc1 的 spuName)
BM25 = IDF × TF部分
= 5.29 × 1.11
≈ 5.87

ES里字符串有两种类型：
● text会被分词用于全文检索，比如用户搜红色短袖T恤，ES把它拆成"红色"、"短袖"、"T恤"，然后去倒排索引里匹配。
● keyword则不会分词，整体作为一个词条，用于精准匹配、聚合、排序，比如分类名”男装上衣“你不希望它被拆成"男装"和"上衣"分别匹配，而是要完整匹配。
所以说spuName需要分词，description需要分词，类型名不能分词，
ES八股
1. ES数据格式
   Index索引类似于数据库的表，做逻辑隔离
   Document为ES中的数据形式，为json
   Field就是json的key
   Mapping映射定义字段类型及索引方式，就是index的表结构定义，定义index有哪些字段，字段的类型是什么，怎么给这个index的document做分词。
   正排索引就是文档->词，倒排就是词->文档
   搜索要的就是倒排，因为输入的是词
2. 倒排索引的结构
   a. 词条字典，也就是说类似红色、纯棉、t恤这些拆分后的关键词
   b. 词条索引，词条字典很大，不可能全量加载在内存，于是就加一个索引，类似于词条字典的前缀树，节省空间，在内存里的是这个词条索引，它定位到这个词条在字典上的位置，去磁盘找
   c. 倒排列表，每个磁盘对应的document列表，不只是ID，还有词频（这个词在这个文档出现几次），这个词在这个文档中的位置。
3. 查询类型
   a. match：对查询词分词，去倒排索引查询
   b. multi_match：同时在多个字段上做match。
   productName 的倒排索引：
   “纯棉” → [doc1, doc3, doc5]
   “羊毛” → [doc2, doc4]

description 的倒排索引：
“纯棉” → [doc1, doc2, doc3]
“耐磨” → [doc4, doc5]
c. term：查询keyword字段
d. range为范围查询
e. bool为组合多个查询条件
4. 深分页问题
   用户翻到100页，每页10条，ES怎么处理？
   ES的分页是协调节点汇总式的，协调节点将请求分发给这个index的各个shard，各个shard根据查询请求将input分词，然后和词条字典进行比对，比如”纯棉T恤“可能命中两个倒排索引，我们就做一个交集，根据ES的BM25相关性得分算法计算每一个文档的相关度，我们要查的是from 990 - 1000，每个shard得把所有文档的相关性得分都算了，最后返回Top 1000，不管你翻多少页，各个shard的打分都是一样的复杂度，但是最后给协调节点做归并排序的时间开销是不一样的，翻得越深开销越大。
   而search_after则是协调节点会把每一次查询的最后一条的相关性得分给算一下，返回回去，用户下一次查询就带着这个最后一条相关性得分来。
   search_after其实是Feed流的最好解法。
   而WEB一般情况下其实还是有页码的，这种我们就限制最大查询页数，100页以内完全不影响用户使用，ES在处理100页以内的性能开销也不大。
   页数问题
   ES每次查询都会返回一个total，就是说有多少文档命中了。
   前端可以拿这个total来计算最大页数。

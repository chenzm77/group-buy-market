## 用户服务（User Service）

### 服务能力

- **拆分依据**：DDD用户上下文，聚合根 `User` + 独立聚合 `Address`，与认证服务分离。
- **提供能力**：本人资料/地址CRUD、管理员查询/列表、内部批量查用户/地址。

### 角色

角色只有两个，我们是自营电商，没有店铺的概念，全局只有admin（商城运营）和user

### user表

```plsql
CREATE TABLE `t_user` (
    `id` BIGINT NOT NULL COMMENT '用户ID，雪花算法生成',
    `phone` VARCHAR(32) NOT NULL COMMENT '手机号',
    `nickname` VARCHAR(32) DEFAULT NULL COMMENT '昵称',
    `avatar` VARCHAR(512) DEFAULT NULL COMMENT '头像URL',
    `role` VARCHAR(32) NOT NULL DEFAULT 'USER' COMMENT '角色：USER/ADMIN',
    `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0-封禁 1-正常',
    `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_phone` (`phone`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

3. RPC内部调用

### 架构说明

本规范适用于所有基于 DDD 的微服务项目，明确各层职责与依赖关系，确保业务逻辑内聚、技术细节解耦。

------

### 一、模块划分

text

```
project-root/
├── xxx-api          # 对外契约：RPC 接口、DTO（可选，视是否需要独立发布）
├── xxx-app          # 应用层：用例编排、事务、权限
├── xxx-domain       # 领域层：业务逻辑、聚合、值对象、仓储接口、领域服务
├── xxx-infrastructure # 基础设施层：实现仓储、RPC 客户端、消息、缓存
├── xxx-trigger      # 触发器层：接收外部请求（Dubbo、MQ、定时任务）
├── xxx-types        # 共享类型：常量、枚举、异常、基类（被所有层依赖）
```



------

### 二、核心依赖原则（编译时）

text

```
app  ──→ domain ←── infrastructure
trigger → app
api    → types
types  (无依赖，被所有层依赖)
```



**关键约束：**

- **domain 不依赖任何其他业务模块**（只依赖 `types` 和 JDK）
- **infrastructure 依赖 domain**（实现其定义的仓储接口）
- **app 只依赖 domain**（不依赖 infrastructure）
- **trigger 依赖 app**（并实现 `api` 接口）
- **运行时**：Spring 将 infrastructure 的实现注入到 domain 或 app 中（依赖倒置）

------

### 三、各层详细职责

#### 1. `types` – 共享类型层

| 内容     | 说明                           |
| :------- | :----------------------------- |
| 常量类   | `Constants`                    |
| 枚举     | `ResponseCode`, `StatusEnum`   |
| 通用异常 | `AppException`, `BizException` |
| 事件基类 | `BaseEvent`                    |
| 工具类   | 可选                           |

**规则**：无任何业务逻辑，可被任意层依赖。

------

#### 2. `api` – 接口契约层（可选）

**用途**：当服务需要对外暴露 RPC 接口时，在此定义 Dubbo/gRPC 的 **interface** 和 **DTO**。

**内容**：

- 服务接口（如 `UserProfileService`）
- 请求参数类（`XxxRequest`）
- 响应结果类（`XxxResponse`, `XxxDTO`）

**依赖**：仅依赖 `types`。

**注意**：不包含任何实现。

------

#### 3. `trigger` – 触发器层（入口适配器）

**作用**：接收外部请求，是系统的 **入口**。

**职责**：

- 实现 `api` 模块中的 RPC 接口
- 监听 MQ、定时任务、HTTP 回调（如网关直连转 Dubbo）
- **只做协议转换、参数校验、身份提取**
- **直接调用 `app` 层的一个方法**（一行委托，不做业务编排）
- 将 `app` 层返回结果转换为 DTO，处理异常并返回错误码

**依赖**：`api`（实现接口）、`app`（调用）、`types`

**示例**：

java

```
@DubboService
public class UserProfileTrigger implements UserProfileService {
    @Resource
    private UserApplicationService userApp;

    @Override
    public UserProfileDTO getProfile(Long userId) {
        return userApp.getUserProfile(userId);
    }
}
```



**禁止**：直接调用 `domain` 或 `infrastructure`。

------

#### 4. `app` – 应用层

**作用**：编排用例，协调领域对象完成业务场景。

**职责**：

- 开启/管理事务（`@Transactional`）
- **调用领域服务（domain service）或聚合根方法**
- 权限校验、日志记录
- 发送领域事件（通常通过领域服务或事件总线）
- **不包含任何业务规则**（规则必须下沉到 domain）

**依赖**：**只依赖 `domain`**（和 `types`）
**不依赖**：`infrastructure`、`trigger`、`api`

**示例**：

java

```
@Service
public class UserApplicationService {
    private final UserDomainService userDomainService;

    public UserApplicationService(UserDomainService userDomainService) {
        this.userDomainService = userDomainService;
    }

    @Transactional
    public void updateUserNickname(Long userId, String nickname) {
        // 只做编排，业务逻辑在 domain
        userDomainService.changeNickname(userId, nickname);
    }
}
```



------

#### 5. `domain` – 领域层（核心）

**作用**：实现所有业务逻辑，是系统的 **心脏**。

**内容**：

- **聚合根（Aggregate Root）**：如 `User`、`Order`
- **实体（Entity）**、**值对象（Value Object）**
- **领域服务（Domain Service）**：处理跨聚合或复杂的业务逻辑
- **仓储接口（Repository Interface）**：定义数据持久化抽象
- **领域事件定义**

**核心原则**：

- **仓储接口定义在此层**，由 `infrastructure` 实现
- **业务规则封装在聚合根或领域服务中**
- **不依赖任何框架**：无 Spring 注解（`@Service` 等可用在领域服务实现上，但接口定义在 domain）、无 MyBatis、无中间件 SDK
- 可依赖 `types` 中的异常、常量等

**示例**：

java

```
// 仓储接口（定义在 domain）
public interface UserRepository {
    User findById(Long id);
    void save(User user);
}

// 领域服务（调用仓储，执行业务逻辑）
@Service  // Spring 注解允许，但实际逻辑在 domain
public class UserDomainService {
    private final UserRepository userRepository;

    public UserDomainService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public void changeNickname(Long userId, String newNickname) {
        User user = userRepository.findById(userId);
        if (user == null) throw new AppException("用户不存在");
        user.updateNickname(newNickname);  // 业务规则在聚合内
        userRepository.save(user);
    }
}

// 聚合根
public class User {
    private Long id;
    private String nickname;

    public void updateNickname(String newNickname) {
        if (newNickname == null || newNickname.isBlank()) {
            throw new AppException("昵称不能为空");
        }
        if (newNickname.length() > 20) {
            throw new AppException("昵称过长");
        }
        this.nickname = newNickname;
    }
}
```



------

#### 6. `infrastructure` – 基础设施层

**作用**：实现 `domain` 层定义的仓储接口，处理所有对外部资源的访问。

**职责**：

- **实现仓储接口**（如 `UserRepositoryImpl`）
- 数据库访问（MyBatis、JPA）
- 调用其他微服务的 RPC 客户端
- 消息发送（MQ Producer）
- 缓存操作（Redis）
- 配置、安全等

**依赖**：

- 依赖 `domain`（**因为要实现其接口**） ✅ 正确依赖
- 依赖 `api`（调用其他服务时）
- 依赖第三方库（MyBatis、Redis client）

**注意**：**infrastructure 不包含任何业务逻辑**，只做技术实现。

**示例**：

java

```
@Repository
public class UserRepositoryImpl implements UserRepository {
    @Autowired
    private UserMapper userMapper;

    @Override
    public User findById(Long id) {
        UserPO po = userMapper.selectById(id);
        if (po == null) return null;
        return new User(po.getId(), po.getNickname());
    }

    @Override
    public void save(User user) {
        userMapper.updateNickname(user.getId(), user.getNickname());
    }
}
```

### 四、完整调用链示例（用户修改昵称）

text

```
外部 Dubbo 请求 
  → UserProfileTrigger (实现 api 接口)
    → UserApplicationService.updateUserNickname (app 层)
      → UserDomainService.changeNickname (domain 领域服务)
        → UserRepository.findById (domain 定义的接口)
          → UserRepositoryImpl.findById (infrastructure 实现)
        → user.updateNickname(...) (domain 聚合根业务方法)
        → UserRepository.save (domain 接口)
          → UserRepositoryImpl.save (infrastructure)
      ← 返回 void
    ← 返回 void
  ← 返回 DTO
```



------

### 五、总结（一句话记忆）

> **app 只调领域服务，domain 写规则 + 定义仓库接口，infra 实现仓库，trigger 薄薄转发。**

**依赖方向（编译时）**：
`trigger` → `app` → `domain` ← `infrastructure`
`api` → `types`，`domain` → `types`，其余层依赖 `types`

**禁止事项**：

- app 直接调用 infrastructure 或数据库
- domain 依赖 infrastructure 或 app
- infrastructure 中包含业务逻辑
- trigger 中编写业务编排

遵循此规范，即可构建清晰、可测试、可维护的 DDD 微服务。
